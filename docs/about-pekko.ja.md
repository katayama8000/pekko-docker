# Pekko とは何か / 何がうれしいのか

このドキュメントは、本サンプル（[README](../README.md)）で使っている
**Apache Pekko** がそもそも何で、どこがうれしいのか、そして同じことを
**Node.js / Rust** でやるとどうなるのかを日本語でまとめたものです。

---

## 1. Pekko とは

**JVM 上で並行・分散・メッセージ駆動なアプリを作るためのツールキット**です。

- 元は **Akka**（Scala 製）。2022 年に Akka がライセンスを商用寄り（BSL）へ
  変更したため、最後の Apache 2.0 版をコミュニティがフォークして
  **Apache Pekko** が生まれた。中身はほぼ Akka で、パッケージ名が
  `akka.*` → `org.apache.pekko.*` に変わったのが主な違い。
- Scala で実装され、**Scala / Java の両方**から使える。
- 主なモジュール:
  - **pekko-actor**（本体: アクターモデル）
  - **pekko-stream**（バックプレッシャ付きストリーム処理）
  - **pekko-http**（HTTP サーバ / クライアント。本サンプルで使用）
  - **pekko-cluster / cluster-sharding**（複数ノードへの分散）
  - **pekko-persistence**（イベントソーシングで状態を永続化）

### 中核は「アクターモデル」

```
状態（state）はアクターの中だけに存在し、外から直接触れない。
やりとりは「不変メッセージを送る」ことだけ。
1つのアクターはメッセージを 1 件ずつ順番に処理する。
```

この結果、**ロック（synchronized / Mutex）を一切使わずに**状態の一貫性が保たれます。
「共有メモリ + ロック」ではなく「メッセージのやり取り」で並行性を扱うのが
発想の転換点です。

---

## 2. 何がうれしいのか

| 特徴 | 中身 | うれしさ |
|------|------|----------|
| **ロックフリーな並行性** | アクターが 1 件ずつ処理 | デッドロック・競合・lost update を構造的に防げる |
| **1 エンティティ = 1 アクター** | 口座・ユーザ・部屋などを軽量アクターで表現 | 数百万個でも扱える（1 アクター ≒ 数百バイト） |
| **位置透過性** | 同じ `ask` で同一ノードでも別ノードでも送れる | スケールアウト時にコードがほぼ変わらない |
| **障害監視（supervision）** | 親が子の例外を検知し再起動戦略を選ぶ | "let it crash"。障害をローカルに封じ込める |
| **永続化（Persistence）** | 状態変更をイベントとして保存・再生 | クラッシュ後に状態を復元（イベントソーシング） |
| **バックプレッシャ（Streams）** | 速い生産者が遅い消費者を溢れさせない | メモリ枯渇を防ぐ |

### 逆に「うれしくない / 不要」な場面

正直に言うと、**本サンプルのような単純な CRUD・低負荷 API では過剰**です。
普通のサーバ + DB トランザクションで十分。Pekko が効くのは:

- 同一エンティティへの**高頻度な並行更新**がある
- **大量の状態持ちエンティティ**を捌く（チャット、ゲーム、IoT、取引）
- **分散・耐障害**が要件
- **ストリーム処理**でバックプレッシャが要る

これらが無いなら、学習コスト分だけ損になり得ます。

## 3. 同じことを Node.js だとどうなるか

### 並行性モデル

Node は **シングルスレッド + イベントループ**。JS のコールバック/await が
途切れない限り処理は割り込まれないので、**単一プロセス内なら**「口座残高を
読んで足して書く」は自然にアトミックで、ロックも要りません。

```js
let balance = 0;
app.post("/deposit", (req, res) => {
  balance += req.body.amount;   // await を挟まなければ競合しない
  res.json({ balance });
});
```

- **似ている点**: 「1 件ずつ処理」という意味ではアクターの直列性に近い。
  軽量 API ならこれで十分で、Pekko より圧倒的に手軽。
- **違う/弱い点**:
  - CPU バウンドな処理で**1 コアしか使えない**（worker_threads や
    cluster で多プロセス化が必要、状態共有は別問題に）。
  - `await` を挟むと**その間に他リクエストが割り込む**ため、
    結局アプリ側で排他制御が要ることがある。
  - **分散・耐障害・永続化（イベントソーシング）・supervision** を
    まとめて面倒見る標準基盤は無い。自前 or 各種ライブラリの寄せ集め。
  - 「1 エンティティ 1 アクター」を数百万個、という発想は標準では持ちにくい。
- アクター志向を Node でやるなら **Comedy**, **Nact**（Akka 風ライブラリ）や、
  分散ステートフルなら **Cloudflare Durable Objects**, **Microsoft Orleans
  系（.NET だが思想が近い）**, **Temporal** などが該当領域。

### まとめ
> Node は「小〜中規模の I/O バウンド API」を最速で作れる。
> ただし**マルチコア活用・分散ステート・耐障害**を本気でやると、
> Pekko が標準で持っている部分を自前で組む必要が出てくる。

## 4. 同じことを Rust だとどうなるか

### 並行性モデル

Rust は **本物のマルチスレッド**で、**所有権と型システムがデータ競合を
コンパイル時に防ぐ**（`Send`/`Sync`）。共有状態は普通こうなる:

```rust
use std::sync::Mutex;
struct Account { balance: Mutex<i64> }

async fn deposit(acc: &Account, amount: i64) -> i64 {
    let mut b = acc.balance.lock().unwrap();  // 明示的にロック
    *b += amount;
    *b
}
```

- **似ている点**: tokio などで非同期、`tokio::sync::mpsc` チャネルで
  「タスクへメッセージを送る」アクター風パターンが書ける
  （**Actix / Coerce / Ractor / Kameo** などのアクターフレームワークもある）。
- **違う/強い点**:
  - **GC が無い**ので低レイテンシ・省メモリ。データ競合は**コンパイラが拒否**。
  - その代わり、共有状態には `Mutex`/`RwLock`/`Arc` を**明示的に書く**のが基本。
    Pekko は「ロックを書かない」、Rust は「ロックを安全に書ける」アプローチ。
  - 分散・クラスタリング・永続化・supervision を**標準で**まとめた成熟基盤は、
    JVM の Pekko ほど揃っていない（個別クレートの組み合わせになりがち）。
- ロックを避けたいなら Rust でも「1 タスク = 1 エンティティ + mpsc チャネル」で
  アクターパターンを採用でき、これは Pekko の思想とほぼ同じ。

### まとめ
> Rust は**性能と安全性**が最強クラスだが、並行制御は基本「明示的なロック」。
> アクター的にやりたければライブラリで実現でき、考え方は Pekko に近い。
> 一方で**分散・耐障害をフルセットで**となると JVM/Pekko のエコシステムが厚い。

## 5. 三者ざっくり比較表

| 観点 | Pekko (JVM) | Node.js | Rust |
|------|-------------|---------|------|
| 並行モデル | アクター（メッセージ駆動） | シングルスレッド + イベントループ | マルチスレッド + 所有権 |
| マルチコア活用 | ◎ 標準 | △ プロセス分割が必要 | ◎ 標準 |
| データ競合の防ぎ方 | アクターが直列化（ロック不要） | そもそも単一スレッド | コンパイラが拒否 / 明示ロック |
| 大量ステートフル実体 | ◎ 1実体1アクター | △ 自前設計 | ○ 自前 or ライブラリ |
| 分散・クラスタリング | ◎ Cluster Sharding | △ 自前/外部 | △ クレート組合せ |
| 耐障害（supervision） | ◎ 標準 | × 自前 | △ ライブラリ |
| 永続化（イベントソーシング）| ◎ Persistence | △ 自前/外部 | △ 自前/外部 |
| 性能・省メモリ | ○（GC あり） | ○ | ◎ |
| 学習コスト・手軽さ | △（概念が多い） | ◎ | △（所有権） |

> ◎=標準で強い / ○=実用十分 / △=可能だが自前寄り / ×=苦手

## 6. このサンプルとの対応

本リポジトリの実装で Pekko の要素がどこにあるか:

- アクター本体・状態・メッセージ protocol → [`BankAccount.scala`](../src/main/scala/com/example/BankAccount.scala)
- `ActorSystem` 起動・アクターの spawn → [`Main.scala`](../src/main/scala/com/example/Main.scala)
- HTTP とアクターの橋渡し（`ask` パターン）→ [`BankRoutes.scala`](../src/main/scala/com/example/BankRoutes.scala)

現状は口座 1 つなので Pekko の旨味は出ていません。価値を体感するには
「1 口座 1 アクター + 同一口座への並列負荷テスト」へ拡張し、
**ロック無しでも残高がズレない**ことを見せるのがおすすめです。
